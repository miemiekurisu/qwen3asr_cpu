# Audit Report: 2026-04-10

## Purpose

This audit documents the current drift between the project design/specification and the actual implementation in `qwen-asr-provider` as of 2026-04-10.

## Scope

- Design documents: `docs/00-reference-analysis.md` through `docs/07-license-compliance.md`
- Source implementation under `include/`, `src/`, `app/`, and `tests/`
- Build and runtime configuration files including `CMakeLists.txt`, `CMakePresets.json`, Docker support, and build scripts
- Test coverage and compliance with `docs/04-development-standard.md`

## Summary of Findings

1. **Architecture drift is major**
   - The design specifies a layered C++ architecture with separate `storage`, `model`, `audio`, `inference`, `runtime`, and `protocol` modules.
   - The actual implementation has only the protocol/service scaffolding in C++ and relies on a C backend in `src/backend/qwen_cpu/` for the core model pipeline.
   - Approximately 82% of the design-specified C++ components are missing.

2. **Missing design components**
   - `qasr/storage/safetensors_loader`: completely absent
   - `qasr/model/tokenizer`: completely absent as a C++ layer
   - `qasr/audio/frontend`: absent
   - `qasr/inference/encoder` / `decoder`: absent as C++ abstractions
   - `qasr/inference/streaming_policy`: absent except for UTF-8 helper logic
   - `qasr/runtime/session_manager` and `task_queue`: absent
   - `qasr/runtime/realtime_session`: struct exists, but core methods are missing
   - `qasr/protocol/vllm_server`: validation only; handler missing

3. **State machine drift**
   - Design-specified session and request state machines are not implemented as explicit enums or state objects.
   - The realtime text state is only partially present via `RealtimeTextState`, without an explicit `Unseen → Partial → Stable → Final` enum.

4. **Test coverage drift**
   - Only 3 out of 52 public functions meet the document-required test categories (normal, extreme, error, random).
   - Critical functions without unit tests: `RunAsr()`, `BuildServerUsage()`, `RunServer()`.
   - Many validation and format functions lack extreme and random-value tests.

5. **Service and protocol drift**
   - OpenAI-compatible endpoints exist, but vLLM routing is not implemented.
   - `/api/realtime/*` exists, but there is no WebSocket realtime endpoint and SSE streaming is only superficially supported.
   - Async job state cleanup is missing, risking unbounded memory use.
   - Request tracing is incomplete: metrics exist, but distributed trace/request_id support is lacking on realtime paths.

6. **Build and runtime drift**
   - BLAS selection is correct: Accelerate on macOS, OpenBLAS on Linux.
   - `CMakeLists.txt` supports ASan/UBSan, but `CMakePresets.json` does not include sanitizer presets.
   - Docker build path is present, but Linux/OpenBLAS full regression has not been confirmed due to apt mirror issues.

## Recommendations

1. **Decide the architectural path**
   - Either update design docs to reflect a C++ protocol / C backend architecture, or begin implementing the missing C++ layers to match the design.

2. **Implement missing C++ components in priority order**
   - `storage`, `model`, `audio`, `inference`, `streaming_policy`, `runtime/session_manager`, `task_queue`.

3. **Augment tests immediately**
   - Add unit tests for `RunAsr()`, `BuildServerUsage()`, and `RunServer()`.
   - Add pseudo-random and extreme-value tests for all public API validation functions.

4. **Improve service robustness**
   - Add explicit async job TTL cleanup.
   - Add request_id support for realtime endpoints.
   - Implement true SSE/WebSocket streaming for realtime and chat.

5. **Fix build presets**
   - Add sanitizer presets to `CMakePresets.json` for ASan/UBSan.
   - Confirm Linux/OpenBLAS Docker build once apt mirrors stabilize.

## Conclusion

The project is functionally moving forward, but the current implementation has significant drift from the intended C++ layered design and from the testing standards defined in `docs/04-development-standard.md`. This audit recommends either a deliberate reprioritization toward the actual implemented architecture or a renewed effort to restore the designed architecture with dedicated module-by-module delivery.
