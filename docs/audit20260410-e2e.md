# E2E 集成测试报告（端到端音频推理）

日期：2026-04-10
测试平台：macOS arm64 (Accelerate) + Docker linux/arm64 (OpenBLAS)

## 一、测试概述

使用 `testfile/` 目录下的真实音频文件，对本地和 Docker 两种环境进行端到端 ASR 推理测试。
同时运行单元测试，验证代码变更后的兼容性。

### 测试音频

| 文件 | 时长 | 原格式 | 内容 |
|------|------|--------|------|
| `短句.mp3` → `short_test.wav` | 2.95s | MP3 16kHz stereo 48kbps | "我这边有人，你能过来救我一下吗？" |
| `顾君子（01）.mp3` → `long_test.wav` | 28:46 | MP3 44.1kHz stereo 32kbps | 有声读物（古典文学） |

音频预处理：`ffmpeg -ar 16000 -ac 1 -f wav` 转换为 16kHz 单声道 WAV。

## 三、发现的漂移问题

### D-1: tokenizer.h 缺失 `<memory>` 头文件（已修复）

- **严重度**：High
- **现象**：`docker build --no-cache` 时 GCC 报错 `'shared_ptr' in namespace 'std' does not name a template type`
- **原因**：`include/qasr/model/tokenizer.h` 使用 `std::shared_ptr` 但未包含 `<memory>`。Clang (macOS) 通过间接包含自动引入，GCC (Linux) 严格要求显式包含。
- **修复**：已添加 `#include <memory>` 到 `tokenizer.h`。
- **设计规范参考**：`docs/04-development-standard.md` 要求跨平台编译一致性。

### D-2: 音频格式仅支持 WAV

- **严重度**：Medium
- **现象**：`testfile/` 下的 MP3 文件无法直接使用，需先用 ffmpeg 转 WAV。
- **原因**：C 后端 `qwen_load_wav()` 只支持 WAV PCM 格式。
- **漂移**：`docs/05-ui-and-docker.md` 和 `testfile/README.md` 列出 MP3 测试文件，但 CLI 不支持直接输入 MP3。
- **建议**：CLI 层面添加 ffmpeg pipe 自动转换，或在文档中明确说明需手动转换。

### D-3: 长音频流式输出质量受限

- **严重度**：Low（音频质量问题，非代码 bug）
- **现象**：28 分钟有声读物仅识别出 2 个文本片段。
- **原因**：原始 MP3 仅 32kbps，含大量背景音乐和低清晰度语音。
- **建议**：`testfile/README.md` 应注明各文件的预期识别质量等级。

### D-4: segment-max-codepoints 依赖不明显

- **严重度**：Low
- **现象**：`ShouldFlushAsrSegment()` 当 `max_codepoints <= 0` 时永不触发刷新。
- **漂移**：设计规范要求默认值 48（`AsrRunOptions` 结构体默认值正确），但 `ShouldFlushAsrSegment` 函数的 0-guard 行为未在文档中说明。
- **建议**：文档补充说明 `0` 表示"仅在强制刷新时输出"。

### D-5: Dockerfile apt-get 偶发失败

- **严重度**：Low
- **现象**：首次 `--no-cache` 构建因 apt-get exit code 100 失败，重试后成功。
- **原因**：Docker Desktop 中 ubuntu:24.04 base image 的 apt 源偶发超时。
- **建议**：Dockerfile 中添加 `--retry 3` 或使用国内镜像源（已在 `docker_linux_openblas.sh` 中实现但 Dockerfile 未同步）。
