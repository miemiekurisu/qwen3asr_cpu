# 开发规范

版本：强制
日期：2026-04-10

## 一、总律

1. 二参考项目只读，不改。
2. 构建、包含、链接，不得直接依赖二参考项目目录。
3. 若需借其代码，必须复制入本仓 `vendor/` 或自研目录后再用。
4. 任一函数，必须有单元测试。
5. 单测必须含正常值、极值、随机值、错误值。
6. 任一模块完成后，必须依次过：单测、集成、回归、文档漂移检查。
7. 首务 CPU；优化以跨平台为前提，不得为单平台牺牲架构。
8. 实时 ASR 不得假设说话间有可用停顿。
9. 你的所有思考过程和回复都强制使用文言文（文档不需要），并且尽量去掉之乎者也以及虚词，去掉所有不必要形容词和长定语，能用文言1个字表达的不允许用多于1个字。

## 二、平台与 BLAS

| 平台 | 规定 |
|---|---|
| macOS | Accelerate |
| Linux | OpenBLAS |
| Windows | OpenBLAS |

禁：

- Linux/macOS/Windows 混用不受控 BLAS
- 将平台特化散落业务代码

必须：

- 由构建层一次选定
- 由运行时可查询当前编译结果

## 三、代码规范

### 语言

- C++17
- RAII 为先
- 避免裸 `new/delete`
- 明确 ownership
- 明确线程边界

### 接口

- public API 必写前置条件、后置条件、线程语义
- 可失败函数须回 `Status`
- 不许吞错
- 错误信息短而可判

### 内存

- 模型权重：只读 mmap
- 大缓冲：arena 或复用池
- session 私有态：显式回收
- 不得隐式长生缓存失控

### 并发

- 模型上下文只读共享
- 会话状态独占修改
- 队列必须有界
- 必须支持取消
- 不得无界创建线程
- realtime session 必须有固定 tick，不得假设客户端或音源会停顿来驱动推进，但也要处理客户端停顿

## 四、测试规范

### 单元测试

每一函数至少含：

- 1 组正常值
- 1 组极值
- 1 组错误值
- 1 组随机值或伪随机循环

### 集成测试

每一模块完成后，至少有：

- 与上下游联调
- 资源释放检查
- 并发压测小样
- 连续无停顿语流小样

### 回归

必须保：

- 短音频 golden
- 长音频 golden
- streaming golden
- timestamp golden
- no-pause streaming golden
- partial/stable/final 语义 golden

## 五、内存泄漏与未定义行为

必须开：

- ASan
- UBSan

Linux 额外：

- valgrind 或等价 leak 检查

并发模块：

- TSan 或最小替代验证
- realtime 模块另查 steady-state RSS

禁：

- 测试未跑即合入
- 已知 leak 先留后补

## 六、文档流程

每次开发，文档同步更新：

1. 设计变更
2. 接口变更
3. 测试覆盖
4. 风险与经验

文档漂移检查顺序：

1. 读实现
2. 对照设计
3. 对照测试
4. 更新经验文档

## 七、模块完工门槛

模块若欲标记“完成”，须同时满足：

- 代码合入
- 单测全过
- 集成测试全过
- 回归全过
- sanitizer 全过
- 文档无漂移
- `experience.md` 已记本轮问题

缺一，则曰“未完成”。

## 八、服务规范

- HTTP handler 不得直接持模型可变态
- 协议转换与推理逻辑分层
- SSE / 流式输出须可中断
- 失败须返回结构化错误
- metrics、trace、request id 必备
- realtime 协议须显式区分 `partial` / `stable` / `final`
- VAD 仅可作加速 hint，不可作唯一切段条件
- 未决尾巴必须有年龄上限与强制冻结策略

## 九、提交前清单

1. 单测过否
2. 集成过否
3. 回归过否
4. sanitizer 过否
5. 文档更新否
6. `experience.md` 已记否
7. 是否触碰参考项目；若触碰，则回退
